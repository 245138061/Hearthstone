# 悬浮层与自动识别约束

这份文档定义悬浮层和自动识别的工程约束。后续新增 UI、识别逻辑或截图策略时，默认按这里执行，不再依赖口头约定。

## 1. 场景优先级

- 悬浮层默认以《炉石传说》酒馆战棋横屏对局为主场景设计。
- 新增识别相关 UI 时，先实现横屏布局，再补竖屏退化。
- 不允许用“宽一点就算横屏”的模糊条件。悬浮层布局分支应优先按 `width > height` 判断。

## 2. 自动识别模式

- 自动识别默认是“手动触发一轮”，不是后台持续扫描。
- 用户点击“开始识别/重新识别”后，启动一次短时识别会话。
- 识别会话应尽量短，只覆盖英雄选择阶段的必要时间窗口。
- 一旦锁定本局 5 个种族，立即停止截图和识别。

## 3. MediaProjection 与耗电边界

- `MediaProjection` 授权令牌必须复用，不允许每次重新弹系统授权框。
- 锁定后禁止维持常驻截图监听。
- 非识别会话期间，不应持续保活 `ImageReader -> Bitmap` 全帧转换链路。
- 如果后续恢复自动门控，也必须先走轻量门控，再决定是否进入正式识别。

## 4. 页面门控规则

- WAITING 阶段优先使用轻量门控，不直接跑完整 OCR 或模板匹配。
- 轻量门控优先使用顶部横幅、小区域颜色指纹、亮度分布或其他低成本特征。
- 只有通过页面门控后，才能进入正式识别。
- 非英雄选择页的默认状态应是 `WAITING`，而不是 `NEEDS_ATTENTION`。

## 5. 识别与锁定规则

- 识别结果不是“稳定的 5 个可用种族”时，不允许锁定。
- 不允许将单次 OCR 偶然识别出的 5 个种族直接视为稳定结果。
- 状态机应优先依赖跨帧/跨尝试投票，而不是单帧命中。
- OCR 可以作为辅助信号和调试信息，但不应长期作为唯一主判定。

## 6. UI 反馈规则

- `SCANNING` 必须提供明显的进行中反馈，优先使用黄色呼吸灯。
- `LOCKED` 必须明确提示“已同步当前种族”。
- 锁定后，主操作按钮收敛为小刷新图标，用于开启下一轮识别。
- `NEEDS_ATTENTION` 文案必须表达“本轮未稳定锁定”，不能误导为固定原因，例如“顶部被遮挡”。

## 7. 手动兜底规则

- 手动长按种族卡切换是保底兜底路径，不能移除。
- 即使自动识别存在，也必须保留“重新识别”入口。
- 自动识别失败时，用户必须可以通过手动切换和重新识别完成本局设置。

## 8. 调试规则

- 识别调试信息必须在横屏悬浮层里可见，不能只出现在竖屏或主面板分支。
- 调试信息至少包含：识别结果、OCR 原文、视口、头部 ROI、最近 dump 路径。
- 新增识别逻辑时，优先保证真机可调试，再追求算法复杂度。

## 9. 变更检查清单

每次修改悬浮层或自动识别时，至少自查以下项目：

- 横屏悬浮层是否先得到完整实现
- 非英雄选择页是否显示为 `WAITING`
- 点击“开始识别”后是否进入短时采集
- 锁定后是否立即停止采集
- 下一把是否可以通过刷新图标重新识别
- 长按手动修正是否仍然可用
- 调试信息在横屏下是否可见
