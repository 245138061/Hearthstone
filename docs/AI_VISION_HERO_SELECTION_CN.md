# AI 识图英雄选择页方案

本文档定义一条替代本地 OCR 的新路线：

- 只在英雄选择页触发一次截图
- 将截图发送给 AI 视觉模型做页面理解
- 返回结构化 JSON
- 本地根据种族和后续可选的英雄信息完成推荐

目标是稳定拿到开局这一张图里的关键信息，而不是做整局实时识别。

当前阶段优先级：

- 第一阶段只要求稳定识别 5 个可用种族
- 英雄选项识别和基于英雄的推荐可以延后接入
- 文档中的 `hero_options` 结构仍然保留，用于后续扩展

## 1. 目标

- 自动识别英雄选择页的 5 个可用种族
- 将识图结果转成稳定的本地数据结构
- 在本地完成推荐，不把“选谁更强”外包给视觉模型

## 2. 非目标

- 不做整局持续截图和持续识别
- 不做对局中隐藏信息识别
- 不依赖 hover、长按或额外游戏内交互
- 不要求视觉模型理解完整对局状态

## 3. 推荐链路

建议按下面顺序实现：

1. 用户进入英雄选择页
2. App 触发一次屏幕截图
3. 本地进行轻量裁剪或直接上传整图
4. AI 视觉模型返回结构化 JSON
5. 本地校验 JSON
6. 本地将 `available_tribes` 映射成域模型
7. 第一阶段先基于策略库刷新阵容筛选
8. 后续再叠加英雄数据和英雄推荐

## 4. 触发规则

- 默认只在英雄选择页触发一次
- 首期可以保留“手动点击识别”入口
- 后续如果页面门控足够稳，再升级成“进入英雄选择页自动触发一次”
- 无论自动还是手动，都不应在一局中重复高频调用视觉模型

## 5. 本地职责与 AI 职责边界

### AI 视觉模型负责

- 判断当前截图是不是英雄选择页
- 识别 5 个可用种族
- 在后续阶段识别当前 2 到 4 个英雄选项
- 在后续阶段标记哪些英雄槽位被锁定

### 本地逻辑负责

- 截图与重试策略
- JSON 校验和容错
- 英文/中文种族名映射
- 第一阶段的种族筛选刷新
- 后续的英雄推荐排序
- UI 展示和状态管理

不建议让视觉模型直接返回“该选谁”。这样会把推荐逻辑绑死在远端提示词上，难以复现和迭代。

## 6. 建议的 JSON 返回结构

视觉模型建议只返回 JSON，不返回解释文字。建议结构如下：

```json
{
  "screen_type": "hero_selection",
  "available_tribes": ["Undead", "Pirate", "Naga", "Quilboar", "Dragon"],
  "hero_options": [
    {
      "slot": 0,
      "name": "钩牙船长",
      "hero_card_id": null,
      "locked": true,
      "armor": 14,
      "confidence": 0.96
    },
    {
      "slot": 1,
      "name": "劫匪之王托瓦格尔",
      "hero_card_id": null,
      "locked": false,
      "armor": 14,
      "confidence": 0.98
    }
  ],
  "confidence": 0.95,
  "model_name": "vision-provider-name",
  "raw_summary": "可选种族为亡灵、海盗、纳迦、野猪人、龙"
}
```

### 字段约束

- `screen_type`
  - `hero_selection`
  - `non_target`
  - `unknown`
- `available_tribes`
  - 应返回 5 个标准化种族名
  - 建议优先使用项目内部 `wireName`，例如 `Undead`、`Pirate`
- `hero_options`
  - 第一阶段允许为空
  - 后续接入英雄识图后，`slot` 从左到右递增
  - `name` 允许为空，但不建议
  - `locked` 用于标记锁住的英雄位
  - `armor` 允许为空
- `confidence`
  - 可选，用于后续灰度发布和调试

## 7. 本地校验规则

视觉结果进入推荐前，至少应通过以下校验：

- `screen_type == hero_selection`
- `available_tribes.size == 5`
- `available_tribes` 去重后仍为 5 个
- 如果 `hero_options` 非空，则数量应在 `2..4`
- 如果 `hero_options` 非空，则 `hero_options.slot` 唯一

如果任一条件失败：

- 不更新当前种族
- 不刷新推荐结果
- 在 UI 中显示“本轮识图未锁定，可重试”

## 8. 本地推荐输入

视觉模型返回结果后，本地推荐层只吃这几项：

- `available_tribes`
- 当前策略库
- 后续接入的英雄强度数据

推荐层不应依赖原始截图，也不应依赖模型输出的自然语言解释。

第一阶段如果还没有英雄数据，本地只需要用 `available_tribes` 刷新阵容筛选，不做英雄级排序。

## 9. 失败与重试

建议只保留短链路重试：

- 单次截图失败：直接重试 1 次
- 视觉模型返回 `non_target`：不进红灯，回到 `WAITING`
- 视觉模型返回结构不合法：记日志并进入 `NEEDS_ATTENTION`
- 连续两次失败：提示用户手动再次触发

## 10. 隐私与网络边界

如果后续接入云端视觉模型，需要明确这几点：

- 上传的是英雄选择页截图，不要上传整局持续截图
- 默认只在用户触发时上传
- 文档和设置页里应明确说明“识图会上传截图到远端模型”
- 后续如需国内可用性，应预留多服务商或代理层

## 11. 与现有 OCR 的关系

- 当前本地 OCR 已证明在该界面上稳定性不足
- 视觉模型方案是替代主方案，不是 OCR 的补丁
- 离线图片调试入口仍可保留，用于后续比较 OCR 与 AI 识图结果

## 12. 最小可交付版本

第一版建议只交付下面这些能力：

- 手动触发一次英雄选择页截图
- 调用视觉模型
- 返回结构化 JSON
- 本地显示识别出的 5 个种族
- 用这 5 个种族刷新阵容筛选
- 后续再接英雄数据和英雄推荐

先把“单次截图 -> 单次推荐”做稳，再考虑自动触发和更复杂的 UI。
